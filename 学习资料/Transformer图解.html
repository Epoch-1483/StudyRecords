Transformer 架构通俗图解

从 Attention 到 GPT —— 理解大语言模型的核心引擎

1 Transformer 是什么？

2017 年 Google 发表论文 "Attention Is All You Need"，提出了 Transformer 架构。它彻底抛弃了之前 RNN/LSTM 的逐步循环结构，用一种叫自注意力（Self-Attention）的机制来理解文本。今天你用的 ChatGPT、GLM、DeepSeek、Kimi 全都是 Transformer 的后代。

想象你在读一篇文章。RNN 的做法是从第一个字读到最后一个字，边读边记；而 Transformer 的做法是一眼扫全文，同时分析每个词和其他所有词的关系——就像你同时请了 N 个助手，每个助手负责盯着一个词，然后互相交流"这个词和哪些词有关系"。

2 第一步：把文字变成数字

计算机不认识汉字，所以第一步是把文本切成Token（词元），再转成高维向量。PaiCLI 里你发给 GLM/DeepSeek 的每条消息，都会先经过这个步骤。

为什么用 Token 而不是逐字？ 因为"排序"作为一个整体比"排"和"序"两个字更有语义价值。现代 Tokenizer（BPE / SentencePiece）会统计最常见的字符组合，在词粒度和字粒度之间找平衡。一个中文字通常是 1~2 个 token，一个英文单词通常是 1~3 个 token。

3 位置编码：让模型知道"顺序"

Transformer 是并行处理所有 token 的——它天生不知道哪个词在前、哪个在后。所以需要给每个位置加一个"位置信号"。

就像在一间教室里，所有学生同时举手发言（并行），老师需要给每个学生编号才知道谁先谁后。位置编码就是给每个 token 发的"座位号"。

最终输入 = Token Embedding + Positional Encoding，两个向量直接相加，一起送进 Transformer 层。

4 自注意力（Self-Attention）：Transformer 的灵魂

这是整个架构最核心的机制。它解决的问题是：对于当前这个 token，句子中其他哪些 token 对它重要？重要程度是多少？

想象你在参加一场会议，每个人说完一句话后，大家互相打分："你说的和我说的有多大关系？" 打完分后，每个人把和自己最相关的几个人的观点加权融合到自己下一轮的发言里。

具体来说，每个 token 同时扮演三个角色：Query（提问者）、Key（被查询者）、Value（信息提供者）。

Attention(Q, K, V) = Softmax( Q × Kᵀ / √dk ) × V

除以 √d 的原因：当向量维度 d 很大时，点积的值会很大，导致 Softmax 把所有概率集中到最大值上（梯度消失）。除以 √d 让方差保持稳定，梯度更健康。

5 多头注意力：多角度看同一句话

一组 Q/K/V 只能捕捉一种关系模式。多头注意力就是并行跑多组 Self-Attention，每组用不同的 Wq/Wk/Wv 矩阵，学到不同的关系模式，最后拼接起来。

典型配置：GPT-3 有 96 层 Transformer，每层 96 个注意力头；GLM-4 大约 60~80 层。头数越多，模型能同时捕捉的关系模式越丰富，但计算量也越大。

6 前馈网络 + 残差连接 + 层归一化

注意力层之后，每个 token 的向量还要过一层前馈网络（FFN）做非线性变换，再通过残差连接和层归一化稳定训练。

残差连接就像"保底机制"：即使注意力层或 FFN 没学到什么有用的东西，原始输入 X 也能通过"快捷通道"直接传到下一层，保证信息不丢失。这就是为什么 Transformer 能堆几十层而不崩溃。

层归一化（LayerNorm）就像"音量调节器"：把每层输出的数值范围拉回到一个稳定的区间，防止越传越大或越传越小。

7 GPT 的自回归生成：一个字一个字蹦出来

PaiCLI 对接的所有模型（GLM、DeepSeek、Kimi、StepFun）都是 Decoder-Only 架构。它们生成文本的方式是自回归：每次只预测下一个 token，然后把预测出来的 token 追加到输入里，再预测下下一个。

Masked Self-Attention 的作用：在 Decoder 中，生成第 N 个 token 时不能"偷看"后面的 token。所以注意力矩阵是一个下三角矩阵——每个位置只能关注它自己和它前面的位置。这就是"Masked"的含义。

8 这和 PaiCLI 有什么关系？

理解 Transformer 能帮你理解 PaiCLI 里的很多设计决策：

📡
SSE 流式输出
模型每生成一个 token
就通过 SSE 推一个 chunk
→ PaiCLI 逐行渲染
🧠
reasoning_content
DeepSeek/Kimi 的思考链
也是 token 序列
→ PaiCLI 需要原样回传历史
📐
Context Window
Attention 的计算量是 O(n²)
→ 上下文越长越慢越贵
→ PaiCLI 需要压缩历史
🔧
Function Calling
模型输出特殊的 tool_call token
→ PaiCLI 解析后执行工具
→ 结果作为 tool_result 回传

为什么上下文管理这么重要？ 因为 Self-Attention 的计算复杂度是 O(n²)（每个 token 都要和其他所有 token 做点积），所以模型的上下文窗口是有上限的（128K / 200K / 1M token）。PaiCLI 的 conversationHistory 压缩机制、/compact 命令、Token 预算控制，全都是为了在这个窗口里塞下最有价值的信息。
