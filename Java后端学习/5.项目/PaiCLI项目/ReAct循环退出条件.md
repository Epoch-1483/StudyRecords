# ReAct 循环退出条件：为什么交给 LLM 而非固定轮数

PaiCLI 的 ReAct 循环把**主退出条件交给模型语义判断**（`hasToolCalls()==false`），把固定轮数降级成**兜底保险丝**（budget），平时根本不指望它当出口。

![485](images/react-exit-condition.svg)

## 一、直接回答

「这一步之后任务到底算不算完成」这件事，**只有模型自己能判断，代码在编译期永远不可能知道**。写死一个 `N` 轮，要么截断合法的长任务、要么逼着模型在简单任务上空转——两种都错。

## 二、写死固定轮数 N 为什么必然失败

任务实际需要的步数**由输入数据决定，事先不可知**。同一个工具，`search_code` 一次可能就够，也可能要检索→读文件→再检索→改代码→验证跑 8 步。一个 `N` 无法适配所有情况：

- **N 太小**（如 5）：真要 12 步的「重构这批文件」任务，第 5 步被强制 `return` → 输出残缺、不可用。
- **N 太大**（如 30）：简单任务 2 步做完，但循环没理由停——模型为「填满轮数」会开始**编造工具调用、凑字数**，浪费 token，还可能误触发 `write_file` / `execute_command` 这类有副作用的工具。

根本矛盾：**你选的任何单一 N，对一些任务太小、对另一些太大，没有「刚好」的那一个。**

## 三、为什么 LLM 能当这个裁判

`conversationHistory` 是贯穿所有轮次的唯一状态载体（见 [[ReAct主循环]]），**每轮 `chat()` 都把完整历史（含之前每轮的 tool 意图 + 观察结果）发给模型**。走到第 k 轮时，模型「眼前」不仅有最初的问题，还有它自己前面做的所有事、拿到的所有观察，**掌握判断「语义上是否足够回答用户」所需的全部信息**。代码做不到——代码只能数「轮数到了没」，看不懂「信息齐了没」。

当模型认为「观察已足够」，它就**不再返回 `tool_calls`**（`hasToolCalls()==false`），走 `Agent.java:199` 的 `else` 分支写最终答案并 `return`。这就是 ReAct 天然的「完成信号」：**模型停 = 任务完**。

## 四、那为什么还要 budget？——它是保险丝，不是开关

把退出完全交给模型有个风险：**模型可能不收敛**（死循环 / 振荡 / 永远觉得「还差一点」）。所以 PaiCLI 留了 `:164` 的 `budget.check()` 作为**兜底安全阀**，只在异常时触发：token 用尽 / 停滞（重复调用）/ 超硬轮数，外加 `CancellationContext` 用户取消。

**正常出口永远是「模型不再调工具」，budget 只是熔断。** 这和 [[Prompt注入防御]] 里的 Agent 安全哲学一致——**把 LLM 当「不可信的决策者」，真正兜底的是执行侧护栏，而不是替它做语义判断**。四阀细节见 [[ReAct循环保险阀]]。

## 一句话总结

固定轮数解决不了「任务该几步」这个本质未知量，写死只会产生截断或空转两种错误；LLM 因为能看到完整历史、唯一具备语义完成度的判断力，所以作为主退出；budget / 轮数只当防失控的保险丝。设计上就是：**信任它的判断力，但用护栏兜住它的边界。**
