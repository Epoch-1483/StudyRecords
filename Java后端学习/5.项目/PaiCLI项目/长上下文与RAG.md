# 长上下文与 RAG

结论：长上下文不会让 RAG 失业，最佳实践是「长上下文 + RAG」混合。

## 为什么 RAG 仍必要（长上下文的短板）
- **成本/延迟**：每轮把几十万 token 当输入计费且变慢；RAG 只注入 top-k。
- **中间丢失**：「Lost in the Middle」——信息塞在长文本中段时准确率下降；RAG 把相关片段顶到最前。
- **噪声稀释**：整库无关代码一起喂，重点被淹；RAG 只给相关方法/类。
- **超窗口**：单体仓库 + 依赖轻松破百万 token；RAG 与上下文解耦可扩展。
- **可溯源**：RAG 返回带文件路径的 chunk，agent 可 `read_file` 精读、少幻觉。

## 什么时候可省 RAG
代码库小到能轻松装进 context 还留余量；或需要跨文件全局结构推理（检索反而切碎依赖图）。

## PaiCLI 的做法
`search_code`（[[RAG语义检索]]）先召回相关 method chunk（2000 字符粒度，非整文件）→ Agent 再 `read_file` 把精确片段注入**长上下文**推理。RAG = 精准的上下文装配器，长上下文 = 想得深，两者分工而非替代。对面试官讲这个点比单纯说「做了 RAG」更老练。
