# 1. PageHelper 使用了 Java 的 拦截器机制（类似"半路拦截"）

```
你的代码执行流程：
		↓
dishMapper.pageQuery() 被调用
		↓
PageHelper 拦截器发现："嘿！前面设置了分页！"
		↓
拦截器修改SQL：在原SQL后面加上 LIMIT 语句
		↓
数据库执行修改后的SQL
		↓
返回分页后的结果
```

```
// 步骤1：先设置分页（告诉它要第几页、每页多少条）
PageHelper.startPage(页码, 每页数量);

// 步骤2：紧接着执行查询（必须是紧挨着的下一条查询语句）
List<XXX> list = mapper.查询方法();

// 结果：自动就分页了！不用自己写LIMIT，不用自己算偏移量
```

**示例：**

假设原始的 SQL 是这样的：

```
SELECT * FROM dish WHERE status = 1
```

当调用 **PageHelper.startPage(2, 10)** 后，再执行查询，**PageHelper** 会自动把 SQL 改写成：

```
SELECT * FROM dish WHERE status = 1 LIMIT 10, 10
--                    ↑ 跳过前10条，取接下来10条（也就是第2页）
```

# 2. `@GlobalTransactional` 注解作用

## 2.1. 🎯 `@GlobalTransactional` 的作用

**一句话总结**：这是一个"分布式事务总指挥"注解，确保跨多个微服务的操作要么**全部成功**，要么**全部失败回滚**。

---

## 2.2. 🌟 生活化类比

### 2.2.1. **类比场景：组织一次旅行团**

想象您是旅行社的**总指挥**，要组织一次旅行，需要协调三个部门：

1. **订单部**：创建旅行订单
2. **财务部**：扣客户的钱（账户服务）
3. **资源部**：预订酒店和机票（库存服务）

#### 2.2.1.1. **没有总指挥的情况**（不加注解）：

```
订单部：✅ 订单创建好了
财务部：✅ 钱也扣了
资源部：❌ 不好意思，酒店满房了！

结果：客户钱被扣了，但旅行没订成 😱 数据不一致！
```

#### 2.2.1.2. **有总指挥的情况**（加 `@GlobalTransactional`）：

```
👔 总指挥说："大家听我指挥，一起行动！"

订单部：✅ 订单创建好了
财务部：✅ 钱也扣了  
资源部：❌ 酒店满房了！

👔 总指挥说："出问题了！所有人撤回操作！"
订单部：🔄 删除订单
财务部：🔄 退钱给客户
资源部：🔄 （本来就没成功，不用回滚）

结果：虽然失败了，但数据是一致的（客户钱回来了）✅
```

---

## 2.3. 💻 代码中的具体作用

### 2.3.1. **在这个方法中**：

```
@GlobalTransactional  // 👈 总指挥上线
public Long create(Order order) {
    // 第1步：在订单数据库创建订单
    orderMapper.insert(order);  
    
    // 第2步：远程调用账户服务，扣钱（在另一个微服务/数据库）
    accountClient.deduct(order.getUserId(), order.getMoney());  
    
    // 第3步：远程调用库存服务，减库存（在另一个微服务/数据库）
    storageClient.deduct(order.getCommodityCode(), order.getCount());
    
    return order.getId();
}
```

这三个操作涉及：

- **3 个不同的微服务**（order-service、account-service、storage-service）
- **3 个不同的数据库**

普通的 `@Transactional` 只能管理**单个数据库**的事务，管不了这种跨多个数据库的情况。

`@GlobalTransactional` **就是为了解决这个问题而生的！**

---

## 2.4. 🔍 工作原理详解

### 2.4.1. **加上这个注解后，Seata 会做什么**：

#### 2.4.1.1. **1️⃣** **生成全局事务 ID（xid）**

```
用户下单 → Seata 生成一个唯一的"身份证号"：xid = "12345-abcde"
```

这个 xid 会传递给所有参与的服务，就像给这次交易发了一个"团队标识"。

#### 2.4.1.2. **2️⃣** **开启全局事务**

```
📢 Seata 宣布："全局事务开始了！所有操作都要听我指挥！"
```

#### 2.4.1.3. **3️⃣** **监控所有操作**

```
orderMapper.insert(...)        ✅ 成功
accountClient.deduct(...)      ✅ 成功（TCC 的 Try 阶段）
storageClient.deduct(...)      ❌ 失败！抛出异常
```

#### 2.4.1.4. **4️⃣** **根据结果决定提交或回滚**

**情况 A：全部成功**

```
📢 Seata 说："大家都成功了！"
   ↓
自动调用所有 TCC 服务的 confirm() 方法
   ↓
订单保留、扣款确认、库存确认 ✅
```

**情况 B：任何一个失败**

```
📢 Seata 说："有人失败了！全体回滚！"
   ↓
自动调用所有 TCC 服务的 cancel() 方法
   ↓
订单删除、钱退回、库存恢复 🔄
```

---

## 2.5. 🆚 对比：`@Transactional` vs `@GlobalTransactional`

|   |   |   |
|---|---|---|
|特性|`@Transactional`|`@GlobalTransactional`|
|**适用范围**|单个数据库|多个数据库/微服务|
|**管理者**|Spring 框架|Seata 框架|
|**能管理远程调用吗**|❌ 不能|✅ 能|
|**使用场景**|单体应用|分布式微服务|
|**类比**|本地指挥官，只管自己部门|全球总指挥，管所有部门|

## 2.6. 🎬 完整执行流程演示

### 2.6.1. **成功场景**：

```
// 用户访问：POST /order/create

@GlobalTransactional  // Seata 开始监控
public Long create(Order order) {
    orderMapper.insert(order);              // ✅ 订单入库
    accountClient.deduct("小明", 200);      // ✅ 扣款成功（TRY）
    storageClient.deduct("商品A", 1);       // ✅ 库存充足
    
    // 方法执行完毕，没有异常
} 
// 👇 Seata 检测到成功，自动触发
// ✅ 调用 account 服务的 confirm() - 确认扣款
// ✅ 整个事务提交
```

### 2.6.2. **失败场景**（库存不足）：

```
@GlobalTransactional  // Seata 开始监控
public Long create(Order order) {
    orderMapper.insert(order);              // ✅ 订单入库
    accountClient.deduct("小明", 200);      // ✅ 扣款成功（TRY）
    storageClient.deduct("商品A", 1);       // ❌ 库存不足，抛异常！
    
    // 因为异常，不会正常返回
}
// 👇 Seata 检测到异常，自动触发
// 🔄 调用 account 服务的 cancel() - 退还 200 元
// 🔄 整个事务回滚
// 最终：订单删除、钱退回，数据保持一致
```

## 2.7. 💡 核心要点总结

### 2.7.1. `@GlobalTransactional` **做了什么**：

1. **开启全局事务**：生成唯一的事务 ID（xid）
2. **协调多个服务**：让所有参与的服务都加入这个事务
3. **统一决策**：

- 全部成功 → 通知所有服务提交（调用 confirm）
- 任一失败 → 通知所有服务回滚（调用 cancel）

4. **保证一致性**：确保跨服务的数据一致性

### 2.7.2. **您需要做的**：

✅ 在需要分布式事务的方法上加上这个注解  
✅ 写好 TCC 的三个方法（deduct、confirm、cancel）  
✅ 剩下的交给 Seata 自动处理

# 3. Spring Assert 断言工具类笔记

**核心逻辑**：如果条件**不满足**，则立即抛出 `IllegalArgumentException`。用于“暴力”拦截非法数据，避免后续代码出现空指针或逻辑错误。

|   |   |   |   |
|---|---|---|---|
|方法|作用|暴力解法示例 (代码)|通俗理解|
|`notNull`|对象不能为空|`Assert.notNull(user, "用户不存在");`|“手里必须有东西”|
|`hasText`|字符串非空且有内容|`Assert.hasText(name, "名字不能为空");`|“说话得有声儿”|
|`isTrue`|表达式结果必须为真|`Assert.isTrue(age > 0, "年龄有误");`|“事实必须是这样”|
|`notEmpty`|集合/数组不能为空|`Assert.notEmpty(list, "列表没数据");`|“篮子里得有鸡蛋”|
|`isNull`|对象必须是空的|`Assert.isNull(obj, "这里不该有数据");`|“这里必须是空气”|

## 3.1. 💡 为什么要用它？

1. **省代码**：一行顶四行（省去 `if (xxx == null) { throw ... }`）。
2. **更清晰**：把“检查逻辑”和“业务逻辑”分开，代码读起来像流水账一样顺畅。
3. **统一处理**：配合全局异常处理，能把报错信息优雅地返回给前端。

## 3.2. ⚠️ 注意事项

- 它属于 **“快速失败” (Fail-fast)** 策略：一旦报错，程序立刻中断，不会执行后面的代码。
- 通常放在方法的**最开头**，用来校验入参或数据库查询结果。

# 4. Spring Security PasswordEncoder

**核心作用**：负责密码的**加密存储**和**匹配校验**。它采用的是“单向加密”，即只能从明文算出密文，不能反向解密。

## 4.1. 基础配置（暴力解法）

在 `SecurityConfig` 中定义一个 Bean，通常使用目前最安全的 **BCrypt** 算法：

```
@Bean
public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder();
}
```

## 4.2. 常用方法对比

|   |   |   |   |
|---|---|---|---|
|场景|方法|示例代码|说明|
|**注册/修改密码**|`encode()`|`String pwd = encoder.encode("123456");`|把明文变成一串乱码（如 `$2a$10$...`），存入数据库。|
|**登录/支付校验**|`matches()`|`encoder.matches("123", dbPwd)`|**比对神器**：拿用户输入的明文去和数据库的密文比对。|

## 4.3. 为什么不用 MD5？

- **MD5**：同样的密码生成的密文永远一样，容易被“彩虹表”破解。
- **BCrypt**：**自带“盐值” (Salt)**。即使两个用户的密码都是 `123456`，它们存进数据库的密文也是完全不同的，安全性极高。

## 4.4. 💡 避坑指南

- **不要重复加密**：在登录校验时，**千万不要**先对输入密码 `encode` 再去比对。直接调用 `matches(明文, 密文)` 即可，它内部会自动处理。
- **性能考虑**：BCrypt 为了安全故意算得很慢。如果项目对性能要求极高且并发大，可以考虑 `Argon2PasswordEncoder`。

## 4.5. 🔍 总结口诀

**注册用** `encode` **变密文，登录用** `matches` **辨真假。**

# 5. RestTemplate 工具类

## 5.1. 一、核心定位

`RestTemplate` 是 Spring 提供的一个用于发送 HTTP 请求的工具类。它的本质是 **"Java 界的浏览器"** ——帮你拼接 URL、处理 JSON 转换、管理连接池，让你只需关注"发什么"和"收什么"。

---

## 5.2. 二、两种核心使用方式对比

### 5.2.1. 方式一：快捷方法（getForObject / postForObject）

**特点**：简单粗暴，直接返回你想要的 Java 对象。

#### 5.2.1.1. 常用方法速查表

|   |   |   |   |
|---|---|---|---|
|动作|HTTP 方法|对应方法|返回值|
|查询|GET|`getForObject()`|目标对象|
|新增|POST|`postForObject()`|目标对象|
|修改|PUT|`put()`|void（无返回）|
|删除|DELETE|`delete()`|void（无返回）|

#### 5.2.1.2. 代码示例

```
// GET 请求：根据 ID 查询用户
String url = "http://user-service/users/{id}";
User user = restTemplate.getForObject(url, User.class, 101);

// POST 请求：新增用户
String url = "http://user-service/users";
User newUser = restTemplate.postForObject(url, userForm, User.class);

// PUT 请求：更新用户
String url = "http://user-service/users/{id}";
restTemplate.put(url, userForm, 101);

// DELETE 请求：删除用户
String url = "http://user-service/users/{id}";
restTemplate.delete(url, 101);
```

**优点**：代码量少，逻辑直观，适合简单场景。

**缺点**：

- 拿不到 HTTP 状态码（对方报错 500 也可能拿到 null）
- 无法处理泛型集合（如 `List<ItemDTO>` 会解析失败）
- 无法自定义请求头

---

### 5.2.2. 方式二：全能方法（exchange）

**特点**：返回 `ResponseEntity<T>` 包装类，包含**状态码 + 响应头 + 响应体**三部分。

```
http://item-service/items?ids={ids}
  ↓        ↓            ↓     ↓
协议    服务名       路径  查询参数（占位符）
```

```
你的代码：http://item-service/items
    ↓
RestTemplate 发现是服务名，不是 IP
    ↓
去注册中心（Nacos/Eureka）查询 item-service 的实际 IP
    ↓
拿到：http://192.168.1.100:8080/items
    ↓
发起真正的 HTTP 请求
```

#### 5.2.2.1. 标准代码模板

```
ResponseEntity<List<ItemDTO>> response = restTemplate.exchange(
    "http://item-service/items?ids={ids}",           // ① URL 模板
    HttpMethod.GET,                                   // ② 请求方法
    null,                                             // ③ 请求实体（Header/Body）
    new ParameterizedTypeReference<List<ItemDTO>>() {}, // ④ 泛型类型引用
    Map.of("ids", "1,2,3")                            // ⑤ URL 参数替换
);

// 获取状态码并判断
if (response.getStatusCode().isError()) {
    throw new RuntimeException("服务异常");
}

// 获取响应体数据
List<ItemDTO> items = response.getBody();
```

#### 5.2.2.2. ResponseEntity 的三大组成部分

```
public class ResponseEntity<T> {
    private HttpStatus statusCode;    // 状态码（200/404/500）
    private HttpHeaders headers;      // 响应头（Content-Type/Token 等）
    private T body;                   // 响应体（你要的数据）
}
```

```
你的代码（cart-service）
    ↓
restTemplate.exchange(
    "http://item-service/items?ids={ids}",  // ① URL 模板
    ...,
    Map.of("ids", "1,2,3")  // ② 参数替换
)
    ↓
RestTemplate 处理：
    1. 去注册中心查 item-service 的实际 IP → 192.168.1.100:8080
    2. 替换占位符 → http://192.168.1.100:8080/items?ids=1,2,3
    ↓
发起 HTTP GET 请求：
    GET /items?ids=1,2,3 HTTP/1.1
    Host: 192.168.1.100:8080
    ↓
item-service 的 ItemController 接收：
    @RequestParam("ids") List<Long> ids → [1, 2, 3]
    ↓
返回 JSON：
    [{"id":1, "name":"手机"}, {"id":2, "name":"电脑"}]
    ↓
RestTemplate 反序列化为 List<ItemDTO>
    ↓
你的代码拿到结果
```

**优点**：

- 能拿到完整 HTTP 响应信息，方便容错处理
- 通过 `ParameterizedTypeReference` 完美解析 List/Map 泛型
- 可以自定义请求头（如携带 Authorization Token）
- 支持所有 HTTP 方法（GET/POST/PUT/DELETE/PATCH）

**缺点**：代码稍长，参数较多。

## 5.3. 三、底层原理与生活类比

### 5.3.1. 1. 为什么会有两种写法？

- `getForObject` 是对 `exchange` 的高层封装
- `exchange` 是最底层的统一入口
- Spring 团队为了照顾不同场景，才衍生出了各种快捷方法

### 5.3.2. 2. 生活案例：取快递的两种方式

#### 5.3.2.1. 场景 A：使用 `getForObject`（邻居帮忙）

你找邻居帮忙："帮我去快递点拿个包裹"。

- 邻居直接给你包裹
- 如果包裹丢了（404）或快递点关门了（500），你可能什么反馈都得不到
- 手里可能拿个空气（null），还不知道发生了什么

#### 5.3.2.2. 场景 B：使用 `exchange`（专业快递员）

你找专业快递员帮忙。快递员回来给你一份**签收报告**：

- **状态**：成功签收（200 OK）✅
- **备注**：包裹外包装有点湿（Response Headers）
- **物品**：你要的 User 对象（Response Body）
- 如果对方服务挂了，快递员会明确告诉你："对方仓库着火（500 Error），今天拿不到"

### 5.3.3. 3. 泛型擦除问题（为什么 List 必须用 exchange？）

#### 5.3.3.1. 问题根源

Java 在编译后会"忘掉"泛型的具体类型（泛型擦除）：

```
// ❌ 错误写法：Jackson 不知道 List 里装的是什么
List list = restTemplate.getForObject(url, List.class);
// 结果：List<LinkedHashMap>，强转报错！

// ✅ 正确写法：用 ParameterizedTypeReference 保留类型信息
ResponseEntity<List<ItemDTO>> response = restTemplate.exchange(
    url, HttpMethod.GET, null,
    new ParameterizedTypeReference<List<ItemDTO>>() {}
);
```

`ParameterizedTypeReference` 相当于给 Jackson 留了一张**"类型小纸条"**，告诉它："别瞎猜，这里面装的是 `ItemDTO`。"

---

## 5.4. 四、开发建议：什么时候用哪种？

|   |   |   |
|---|---|---|
|场景|推荐写法|原因|
|返回单个对象，且确定服务稳定|`getForObject`|代码最简洁|
|返回 `List` / `Map` 集合|`exchange`|解决泛型解析报错|
|需要判断对方是否报错（500/404）|`exchange`|通过 `getStatusCode()` 拦截异常|
|需要发送自定义 Header（如 Token）|`exchange`|可以传入 `HttpEntity` 设置请求头|
|发送 PUT/DELETE 请求|`put()` / `delete()`|语义清晰，无需处理返回值|

## 5.5. 五、常见坑点与注意事项

### 5.5.1. 1. URL 参数拼接

```
// ❌ 不推荐：手动拼接，容易出错
String url = "http://item-service/items?ids=" + ids;

// ✅ 推荐：使用占位符 + Map
String url = "http://item-service/items?ids={ids}";
Map<String, Object> params = Map.of("ids", ids);
restTemplate.exchange(url, HttpMethod.GET, null, typeRef, params);
```

### 5.5.2. 2. 中文乱码问题

如果 URL 中包含中文参数，需要编码：

```
String keyword = URLEncoder.encode("手机", "UTF-8");
String url = "http://item-service/search?keyword=" + keyword;
```

### 5.5.3. 3. 超时设置（生产环境必配）

默认情况下 `RestTemplate` 没有超时限制，如果对方服务卡死，你的线程会被一直占用：

```
@Configuration
public class RestTemplateConfig {
    
    @Bean
    public RestTemplate restTemplate() {
        // 创建 HTTP 客户端工厂
        HttpComponentsClientHttpRequestFactory factory = 
            new HttpComponentsClientHttpRequestFactory();
        
        // 设置连接超时：3 秒
        factory.setConnectTimeout(3000);
        // 设置读取超时：5 秒
        factory.setReadTimeout(5000);
        
        return new RestTemplate(factory);
    }
}
```

## 5.6. 六、总结口诀

```
简单查询用 Object，集合泛型用 Exchange。
状态校验头信息，底层全能 Exchange。
PUT  DELETE 直接用，超时配置别忘记。
```

---

**适用场景回顾**：

- 你之前写的 `getForObject`：适合单体项目或确定服务稳定的简单调用
- 项目中用的 `exchange`：适合微服务架构，需要容错处理、泛型解析、状态码校验的复杂场景

# 6. `@ConditionalOnClass` 注解的作用

这个注解的作用是：**"只有当某个类存在时，才加载这个配置类"**。

## 6.1. 结合代码分析

```
@Configuration                      // 这是一个配置类
@ConditionalOnClass(DispatcherServlet.class)  // 只有存在 DispatcherServlet 时才加载
public class MvcConfig implements WebMvcConfigurer {
    
    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        registry.addInterceptor(new UserInfoInterceptor());  // 添加用户信息拦截器
    }
}
```

## 6.2. 为什么要用这个注解？

### 6.2.1. 原因 1：`hm-common` 模块被多个服务引用

```
hm-common 被这些服务引用：
  ├── item-service（商品服务）      → 需要 Web 功能，有 DispatcherServlet ✅
  ├── cart-service（购物车服务）    → 需要 Web 功能，有 DispatcherServlet ✅
  ├── user-service（用户服务）      → 需要 Web 功能，有 DispatcherServlet ✅
  ├── hm-gateway（网关服务）        → 不需要 DispatcherServlet ❌
  └── hm-api（API 客户端模块）     → 不需要 DispatcherServlet ❌
```

### 6.2.2. 原因 2：避免报错

```
如果没有 @ConditionalOnClass：
→ hm-gateway 启动时，会尝试加载 MvcConfig
→ MvcConfig 依赖 DispatcherServlet
→ 但网关服务用的是 WebFlux，没有 DispatcherServlet
→  报错：ClassNotFoundException，启动失败！

有了 @ConditionalOnClass：
→ hm-gateway 启动时，检查是否有 DispatcherServlet
→ 发现没有这个类
→ ✅ 跳过 MvcConfig，不加载，不会报错
```

## 6.3. 底层实现原理

```
Spring Boot 启动流程：
1. 读取 spring.factories
        ↓
2. 发现 MvcConfig 需要加载
        ↓
3. 检查 @ConditionalOnClass(DispatcherServlet.class)
        ↓
4. 扫描 classpath，查找 DispatcherServlet 类
        ↓
5. 判断结果：
✅ 找到了 → 加载 MvcConfig 配置
❌ 没找到 → 跳过 MvcConfig，不加载
        ↓
6. 继续启动...
```

## 6.4. 实际场景举例

**场景 A：启动 item-service（Web 服务）**

```
item-service 的依赖：
    - spring-boot-starter-web  ← 包含 DispatcherServlet

检查：classpath 中有 DispatcherServlet 吗？
    → 有 ✅

结果：加载 MvcConfig，注册 UserInfoInterceptor 拦截器
```

**场景 B：启动 hm-gateway（网关服务）**

```
hm-gateway 的依赖：
    - spring-cloud-gateway  ← 基于 WebFlux，没有 DispatcherServlet

检查：classpath 中有 DispatcherServlet 吗？
    → 没有 ❌

结果：跳过 MvcConfig，不加载拦截器（网关有自己的过滤器机制）
```

## 6.5. 总结

**这个注解的核心作用**：

- ✅ **按需加载**: 只有需要 Web 功能的服务才配置 MVC 拦截器
- ✅ **避免冲突**: 非 Web 服务（如网关）不会加载不相关的配置
- ✅ **提高兼容性**: `hm-common` 可以被各种类型的服务安全引用
- ✅ **防止报错**: 避免 `ClassNotFoundException` 导致启动失败

### 8.2.3. 第3步：网关接收请求，校验 Token

**核心代码**：`AuthGlobalFilter.java`

```
public class AuthGlobalFilter implements GlobalFilter, Ordered {
    
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        // 1. 获取请求
        ServerHttpRequest request = exchange.getRequest();
        
        // 2. 判断是否需要登录（检查白名单）
        if (isExclude(request.getPath().toString())) {
            return chain.filter(exchange);  // 不需要登录，直接放行
        }
        
        // 3. 从请求头获取 Token
        String token = request.getHeaders().getFirst("Authorization");
        
        // 4. 解析并校验 Token
        Long userId = jwtTool.parseToken(token);  // 如果失败会抛出异常
        
        // 5. ✅ 关键：将用户ID写入请求头，传递给下游服务
        String userInfo = userId.toString();
        ServerWebExchange swe = exchange.mutate()
                .request(builder -> builder.header("user-info", userInfo))
                .build();
        
        // 6. 继续转发请求
        return chain.filter(swe);
    }
}
```

**生活类比**：

- 小区保安检查你的门禁卡

### 8.2.5. 第5步：微服务接收请求，拦截器提取用户信息

**核心代码**：`UserInfoInterceptor.java`

```
public class UserInfoInterceptor implements HandlerInterceptor {
    
    @Override
    public boolean preHandle(HttpServletRequest request, 
                            HttpServletResponse response, 
                            Object handler) {
        // 1. 从请求头获取用户信息
        String userInfo = request.getHeader("user-info");
        
        // 2. 如果不为空，存入 ThreadLocal
        if (StrUtil.isNotBlank(userInfo)) {
            UserContext.setUser(Long.valueOf(userInfo));
        }
        
        return true;  // 放行，继续执行Controller
    }
    
    @Override
    public void afterCompletion(HttpServletRequest request, 
                               HttpServletResponse response, 
                               Object handler, 
                               Exception ex) {
        // 3. 请求完成后，清理 ThreadLocal（防止内存泄漏）
        UserContext.removeUser();
    }
}
```

}
```

## 8.3. 三、核心技术点解析

### 8.3.1. 1. ThreadLocal 的作用

```
public class UserContext {
    private static final ThreadLocal<Long> tl = new ThreadLocal<>();
    
    public static void setUser(Long userId) {
        tl.set(userId);  // 存入当前线程
    }
    
    public static Long getUser() {
        return tl.get();  // 从当前线程获取
    }
    
    public static void removeUser() {
        tl.remove();  // 清理，防止内存泄漏
    }
}
```

**通俗解释**：

```
ThreadLocal 就像每个服务员有一个专属笔记本：

服务员A（线程A）的笔记本：用户ID = 123
服务员B（线程B）的笔记本：用户ID = 456
服务员C（线程C）的笔记本：用户ID = 789

服务员A 只能看到自己的笔记本，不会混淆！
```

**为什么必须用 ThreadLocal？**

- Servlet 是单实例多线程的
- 多个请求并发处理，每个请求对应一个线程
- 必须用 ThreadLocal 隔离不同请求的用户信息

---

### 8.3.2. 2. 为什么要在网关层提取用户信息？

**方案对比**：

|   |   |   |   |
|---|---|---|---|
|方案|做法|优点|缺点|
|**方案A：每个服务自己解析 Token**|每个微服务都写 Token 解析代码|服务独立|❌ 代码重复  <br>❌ 每个服务都要依赖 JWT 库|
|**方案B：网关统一解析** ✅|网关解析一次，传用户ID|✅ 代码只写一次  <br>✅ 微服务更轻量|网关性能要求高|

**生活类比**：

```
方案A：每个店铺都配一个验卡机（浪费资源）
方案B：大门统一验卡，登记房间号（高效）
```

### 8.3.3. 3. Gateway 的 mutate() 方法

```
ServerWebExchange swe = exchange.mutate()
        .request(builder -> builder.header("user-info", userInfo))
        .build();
```

**底层原理**：

```
HTTP 请求头本质上是字节流：

原始请求：
  GET /carts HTTP/1.1\r\n
  Host: localhost:8080\r\n
  Authorization: eyJhbG...  ← 原始Token
  
mutate 后的请求：
  GET /carts HTTP/1.1\r\n
  Host: localhost:8080\r\n
  user-info: 123  ← 新增用户ID
  Authorization: eyJhbG...  ← Token 还在（可以删除）
```

**为什么不能直接修改？**

- HTTP 请求是不可变的（Immutable）
- 需要创建一个新的请求对象（类似字符串的不可变性）

## 8.6. 六、常见问题 & 注意事项

### 8.6.1. ⚠️ 问题1：为什么要在 `afterCompletion` 清理 ThreadLocal？

```
@Override
public void afterCompletion(...) {
    UserContext.removeUser();  // 必须清理！
}
```

**原因**：

```
Tomcat 使用线程池处理请求：

请求1 → 线程A → 设置 userId=123
请求2 → 线程A（复用） → 如果不清理，还是 userId=123！
         → 导致用户A看到了用户B的数据！😱
```

**生活类比**：酒店退房后必须换床单，否则下一个客人会用到上一个客人的床单！

---

### 8.6.2. ⚠️ 问题2：`@ConditionalOnClass(DispatcherServlet.class)` 的作用？

```
@Configuration
@ConditionalOnClass(DispatcherServlet.class)
public class MvcConfig { ... }
```

**原因**：

```
hm-common 被多个服务引用：
  ✅ item-service（Web服务）→ 有 DispatcherServlet → 加载拦截器
  ❌ hm-gateway（网关服务）→ 用 WebFlux，没有 DispatcherServlet → 不加载
  
如果没有这个注解：
  → 网关启动时也会尝试加载拦截器
  → 发现没有 DispatcherServlet 类
  → 报错：ClassNotFoundException 💥
```

---

### 8.6.3. ⚠️ 问题3：为什么用 Header 传递而不是 URL 参数？

```
// ✅ 推荐：请求头传递
request.header("user-info", userId)

// ❌ 不推荐：URL 参数
/carts?userId=123
```

**原因**：

1. **安全性**：URL 参数会记录在浏览器历史、服务器日志中
2. **规范性**：用户信息属于请求元数据，应该放在 Header 中
3. **简洁性**：URL 更干净，不需要每个接口都加 `?userId=xxx`

                // 1. 从 ThreadLocal 获取当前用户ID
                Long userId = UserContext.getUser();
                
                // 2. 如果不为空，添加到请求头
                if (userId != null) {
                    template.header("user-info", userId.toString());
                }
            }
        };
    }
}
```

**通俗解释**：

```
RequestInterceptor 的作用：
就像给所有对外电话加一个"自动报号"功能

你打电话给其他部门：
  普通方式：喂，我是张三，帮我查一下...
  拦截器方式：
    → 你只需要说：帮我查一下...
    → 拦截器自动在前面加：我是张三，...
    
代码中的流程：
  你调用：itemClient.queryItemByIds(ids)
    ↓
  拦截器自动添加：Header: user-info=123
    ↓
  实际发送的 HTTP 请求：
    GET /items?ids=1,2,3
    Header: user-info=123  ← 自动添加的！
```

## 9.5. 五、关键技术点详解

### 9.5.1. 1. RequestInterceptor 的工作原理

```
public interface RequestInterceptor {
    void apply(RequestTemplate template);
}
```

**底层实现**：

```
Feign 发起请求的流程：

你调用接口方法
  ↓
Feign 代理对象拦截调用
  ↓
创建 RequestTemplate 对象（请求模板）
  ↓
调用所有 RequestInterceptor.apply(template)  ← 你的拦截器在这里执行
  ↓
从 template 构建实际的 HTTP 请求
  ↓
发送 HTTP 请求
```

**RequestTemplate 是什么**：

```
// RequestTemplate 是一个请求模板对象
template.header("user-info", "123");      // 添加请求头
template.query("ids", "1,2,3");           // 添加查询参数
template.body("{\"name\":\"张三\"}");      // 添加请求体
```

**类比**：

```
RequestTemplate 就像一张"快递单"：
  - 收件人地址 → URL
  - 备注信息 → Header
  - 包裹内容 → Body
  
拦截器就是在快递单上"自动填写备注"的功能
```

---

### 9.5.2. 2. ThreadLocal 的传递性

```
关键问题：为什么 Feign 拦截器能拿到 userId？

线程执行流程：
  Tomcat 线程（假设是线程A）
    ↓
  处理 HTTP 请求
    ↓
  UserInfoInterceptor 存入 ThreadLocal（线程A）
    ↓
  Controller 调用 itemClient.queryItemByIds()
    ↓
  Feign 拦截器从 ThreadLocal 读取（还是线程A）
    ↓
  成功拿到 userId！
```

**为什么能拿到**：

- Feign 调用是**同步**的
- 在同一线程内执行
- ThreadLocal 在同线程内共享

**️ 注意**：

```
// ❌ 如果在异步线程中，ThreadLocal 会丢失！
CompletableFuture.runAsync(() -> {
    // 这里是新线程，拿不到 userId！
    Long userId = UserContext.getUser();  // null
});

// ✅ 正确做法：在主线程获取，传递给子线程
Long userId = UserContext.getUser();
CompletableFuture.runAsync(() -> {
    // 使用传入的 userId
    doSomething(userId);
});
```

|   |   |   |
|---|---|---|
|特性|Gateway 传递|Feign 传递|
|**使用场景**|外部请求 → 微服务|微服务 → 微服务|

**类比**：

```

## 9.9. 九、常见问题

### 9.9.1. ⚠️ 问题1：Feign 调用时 userId 为 null

  1. 请求没有经过网关（直接访问微服务）

**排查步骤**：

```

    if (userId != null) {
        template.header("user-info", userId.toString());
    }
}
```

### 9.9.2. ️ 问题2：忘记配置 defaultConfiguration

### 9.9.3. ⚠️ 问题3：异步调用丢失用户信息

**错误示例**：

```
// 主线程

// 异步线程
CompletableFuture.runAsync(() -> {
    Long uid = UserContext.getUser();  //  null！
    itemClient.queryItemByIds(ids);    // 不会传递用户信息
});
```

**解决方案**：

```

CompletableFuture.runAsync(() -> {
    // 手动设置到子线程
    UserContext.setUser(userId);
    try {
        itemClient.queryItemByIds(ids);
    } finally {
        UserContext.removeUser();  // 清理
    }
});
```

# 10. 动态路由在微服务项目中的作用

动态路由的核心作用是：**不重启网关，就能实时更新路由规则**。

---

## 10.1. 一、通俗类比：商场的"导购牌"

**静态路由**（写在 `application.yaml` 里）：

```
就像商场门口挂着一块"铁制导购牌"，上面刻着：
  - 1楼：服装区
  - 2楼：餐饮区
  - 3楼：电影院

如果商场要新增一个"健身房"在 4楼：
  ❌ 必须把整块铁牌拆下来重新刻
  ❌ 停业半天重新安装
  ❌ 成本高、效率低
```

**动态路由**（从 Nacos 读取）：

```
就像商场用的是"电子显示屏"，导购信息存在后台系统里：
  - 1楼：服装区
  - 2楼：餐饮区
  - 3楼：电影院

如果新增"健身房"在 4楼：
  ✅ 管理员在后台点几下鼠标
  ✅ 电子屏立刻更新
  ✅ 商场不用关门，顾客无感知
```

## 10.2. 二、在你的代码中是怎么工作的？

从你的 `DynamicRouteLoader.java` 可以看到完整流程：

```
@PostConstruct
public void initRouteConfigListener() throws NacosException {
    // 1. 启动时从 Nacos 拉取一次路由配置
    String configInfo = nacosConfigManager.getConfigService()
            .getConfigAndSignListener(dataId, group, 5000, new Listener() {
                
                // 2. 监听配置变化
                @Override
                public void receiveConfigInfo(String configInfo) {
                    // 3. 配置变了，自动更新路由表
                    updateConfigInfo(configInfo);
                }
            });
    
    // 4. 第一次读取也要更新
    updateConfigInfo(configInfo);
}
```

**底层流程**：

```
1. 网关启动
        ↓
2. 从 Nacos 读取 gateway-routes.json
{
    "id": "item",
    "uri": "lb://item-service",
    "predicates": [{"name": "Path", "args": {"pattern": "/items/**"}}]
}
        ↓
3. 解析 JSON，转为 RouteDefinition 对象
        ↓
4. 调用 RouteDefinitionWriter.save() 注册到网关
        ↓
5. 用户访问 /items/123 → 网关匹配到路由 → 转发到 item-service
        ↓
6. 运维人员在 Nacos 控制台修改配置（新增 cart-service 路由）
        ↓
7. Nacos 推送配置变更 → Listener.receiveConfigInfo() 被触发
        ↓
8. 删除旧路由 → 写入新路由
        ↓
9. 新路由立即生效，无需重启网关！
```

## 10.3. 三、静态路由 vs 动态路由对比

|   |   |   |
|---|---|---|
|特性|静态路由（application.yaml）|动态路由（Nacos 配置中心）|
|**配置位置**|写死在代码里|存在 Nacos 配置中心|
|**修改方式**|改代码 → 重新打包 → 重启网关|在 Nacos 控制台修改 → 立即生效|
|**停机时间**|需要重启，服务中断几分钟|零停机，用户无感知|
|**适用场景**|开发阶段、路由规则很少变|生产环境、频繁调整路由|
|**维护成本**|每次改路由都要发版|运维人员直接改配置|

## 10.4. 四、动态路由的核心价值

### 10.4.1. 1. **灰度发布**

```
场景：你要上线一个新版本的商品服务（item-service-v2）

动态路由可以这样操作：
  第一步：90% 流量 → item-service（老版本）
          10% 流量 → item-service-v2（新版本）
  
  第二步：观察新版本有没有 bug
  
  第三步：没问题 → 100% 流量切到新版本
  
  第四步：有问题 → 立刻切回老版本
  
整个过程不用重启网关，秒级切换！
```

### 10.4.2. 2. **服务降级**

```
场景：双十一流量暴增，支付服务扛不住了

动态路由可以这样操作：
  正常情况：
    /pay-orders/** → lb://pay-service（支付服务）
  
  降级时（在 Nacos 改配置）：
    /pay-orders/** → lb://pay-service-backup（备用服务）
  
  或者直接返回友好提示：
    /pay-orders/** → uri: forward:/fallback/pay-error
```

### 10.4.3. 3. **多环境切换**

```
场景：你有开发、测试、生产三套环境

动态路由配置：
  开发环境 Nacos：
    /items/** → lb://item-service-dev
  
  测试环境 Nacos：
    /items/** → lb://item-service-test
  
  生产环境 Nacos：
    /items/** → lb://item-service-prod
  
切换环境只需改 Nacos 配置，不用改代码！
```

## 10.5. 五、代码中的关键逻辑

```
public void updateConfigInfo(String configInfo){
    // 1. 解析 Nacos 传来的 JSON
    List<RouteDefinition> routeDefinitions = JSONUtil.toList(configInfo, RouteDefinition.class);

    // 2. 删除旧路由（防止重复）
    for (String routeId : routeIds) {
        writer.delete(Mono.just(routeId)).subscribe();
    }
    routeIds.clear();

    // 3. 写入新路由
    for (RouteDefinition routeDefinition : routeDefinitions) {
        writer.save(Mono.just(routeDefinition)).subscribe();
        routeIds.add(routeDefinition.getId());
    }
}
```

**为什么先删后加？**

```
想象你在黑板上写课表：
  旧课表：
    周一：语文
    周二：数学
  
  新课表：
    周一：英语
    周三：物理
  
  如果直接写新的：
    黑板上会有：
      周一：语文  ← 旧的没删
      周一：英语  ← 新的
      周二：数学  ← 旧的还在
      周三：物理  ← 新的
  
  正确做法：
    1. 先擦掉整块黑板（删除旧路由）
    2. 再写新课表（写入新路由）
```

## 10.6. 六、总结

**动态路由的本质**：

```
把"写死在代码里的路由规则"变成"可以实时修改的配置"

就像：
  静态路由 = 印刷好的纸质地图（改不了）
  动态路由 = 手机导航软件（随时更新路况）
```

**核心价值**：

- ✅ **零停机更新**：改路由不用重启网关
- ✅ **灵活调度**：灰度发布、服务降级、流量控制
- ✅ **集中管理**：所有路由规则存在 Nacos，一目了然
- ✅ **快速响应**：出问题秒级切换，不用等发版

对于生产环境来说，动态路由是**必备功能**，没有它，每次调整路由都要停机发版，用户体验会非常差！

## 13.3. 💡 关键技术点解析

### 13.3.1. **1. 为什么Token中只存UUID，不存完整用户信息？**

**原因：**

- **安全性**：JWT可以被解码（虽然不能篡改），如果存了敏感信息会泄露
- **灵活性**：用户信息变化时（如修改权限），只需更新Redis，不需要重新发Token
- **大小限制**：JWT太大影响性能

**类比：**

- Token就像**储物柜钥匙**，钥匙上只有一个编号
- 真正的物品（用户信息）存在储物柜（Redis）里
- 每次用钥匙去储物柜取东西

## 13.4. 1️⃣ JWT Token的三部分详解

让我们用**快递包裹**来类比JWT的三部分：

### 13.4.1. **Header（头部）- 快递单的基本信息**

```
{
  "alg": "HS512",    // 使用的加密算法
  "typ": "JWT"       // 令牌类型
}
```

**作用：** 告诉接收方"这是什么类型的令牌，用什么算法签名的"**类比：** 快递单上写着"这是顺丰快递，需要签收"

### 13.4.2. **Payload（载荷）- 包裹里的物品清单**

```
{
  "login_user_key": "abc123-def456-ghi789"  // UUID
}
```

**作用：** 存储实际的数据（在这个项目中只存UUID）**类比：** 包裹里装的东西清单

**注意：** Payload可以被解码（Base64编码），所以**不能放密码等敏感信息**！

### 13.4.3. **Signature（签名）- 快递公司的防伪印章**

```
签名 = HMACSHA512(
  base64UrlEncode(header) + "." + 
  base64UrlEncode(payload),
  secret密钥
)
```

**作用：**

- 验证Token是否被篡改
- 确保Token是服务器颁发的

**类比：** 快递单上的防伪二维码，扫描后能验证这个快递是不是真的顺丰发的

**完整流程：**

```
原始数据 → Base64编码 → 三部分用点号连接 → 生成签名
   ↓
eyJhbGciOiJIUzUxMiJ9.eyJsb2dpbl91c2VyX2tleSI6IjEyMyJ9.xxxxxx
   ↓         ↓                    ↓                    ↓
 Header   点号分隔            Payload              Signature
```

**验证过程：**

1. 收到Token后，用同样的算法和密钥重新计算签名
2. 如果计算出的签名和Token中的签名一致 → Token有效
3. 如果不一致 → Token被篡改过，拒绝接受

15分钟   → 访问接口，剩余15分钟 > 20分钟，不刷新
25分钟   → 访问接口，剩余5分钟 < 20分钟，✅ 自动刷新！
          → Redis中过期时间重置为30分钟（从当前时间算起）
40分钟   → 访问接口，剩余15分钟 > 20分钟，不刷新
55分钟   → 访问接口，剩余5分钟 < 20分钟，✅ 再次刷新！
...
只要用户持续活跃，Token就不会过期
如果用户停止操作超过30分钟，Token才会真正过期
```

**代码流程：**

```
// 每次请求都会经过这个过滤器
JwtAuthenticationTokenFilter.doFilterInternal() {
    LoginUser loginUser = tokenService.getLoginUser(request);
    
    if (loginUser != null) {
        // 👇 这里检查并可能刷新Token
        tokenService.verifyToken(loginUser);
        
        // 设置认证信息
        SecurityContextHolder.getContext().setAuthentication(...);
    }
}

// verifyToken方法
public void verifyToken(LoginUser loginUser) {
    long expireTime = loginUser.getExpireTime();      // 例如：17:30过期
    long currentTime = System.currentTimeMillis();     // 例如：17:15
    
    // 如果剩余时间 <= 20分钟
    if (expireTime - currentTime <= 20 * 60 * 1000) {
        refreshToken(loginUser);  // ✅ 刷新Token
    }
}

// refreshToken方法
public void refreshToken(LoginUser loginUser) {
    // 重新设置登录时间为当前时间
    loginUser.setLoginTime(现在的时间);
    
    // 重新设置过期时间 = 当前时间 + 30分钟
    loginUser.setExpireTime(现在的时间 + 30分钟);
    
    // 更新Redis中的过期时间
    redisCache.setCacheObject(key, loginUser, 30, TimeUnit.MINUTES);
}
```

**好处总结：**

1. **用户体验好** - 正在工作的用户不会被突然踢出
2. **安全性高** - 长时间不活动的用户会自动退出
3. **无需前端配合** - 后端自动处理，前端无感知

## 13.7. 5️⃣ 普通Token vs 长效Token的区别

让我对比说明：

|   |   |   |
|---|---|---|
|对比项|普通Token|长效Token（SSO）|
|**使用场景**|用户名密码登录|企业微信/OA单点登录|
|**有效期**|30分钟（可配置）|7天|
|**JWT中是否包含过期时间**|❌ 不包含|✅ 包含（`.setExpiration()`）|
|**Redis过期时间**|30分钟|7天（10080分钟）|
|**生成方法**|`createToken()`|`createLongTimeToken()`|
|**调用位置**|`login()`|`loginSso()`|

### 13.7.1. **代码对比：**

**普通Token：**

```
// TokenService.createToken()
private String createToken(Map<String, Object> claims) {
    String token = Jwts.builder()
        .setClaims(claims)                    // 只设置载荷
        .signWith(generateKey(secret), HS512) // 签名
        .compact();
    return token;
    // ⚠️ 注意：没有设置过期时间，依赖Redis控制
}

// Redis存储30分钟
redisCache.setCacheObject(userKey, loginUser, 30, TimeUnit.MINUTES);
```

**长效Token：**

```
// TokenService.createLongTimeToken()
private String createLongTimeToken(Map<String, Object> claims) {
    // 👇 计算7天后的时间
    Date expirationDate = new Date(System.currentTimeMillis() + (24*3600*1000));
    
    String token = Jwts.builder()
        .setClaims(claims)
        .setExpiration(expirationDate)       // ✅ 设置了JWT自身的过期时间
        .signWith(generateKey(secret), HS512)
        .compact();
    return token;
}

// Redis存储7天
redisCache.setCacheObject(userKey, loginUser, 7 * 1440, TimeUnit.MINUTES);
```

### 13.7.2. **为什么长效Token要在JWT中也设置过期时间？**

**原因：**

- **双重保险** - JWT自身有过期时间，即使Redis数据丢失，JWT也会过期
- **离线验证** - 某些场景下可以不查Redis，直接验证JWT的过期时间
- **安全性** - 防止Token被无限期使用

**类比：**

- **普通Token** = 临时通行证，只在门卫处（Redis）登记，离开园区就失效
- **长效Token** = 正式员工卡，卡片本身有有效期（JWT过期时间），同时在系统里也有记录（Redis）

# 14. @MdcDot 注解的作用详解

这是一个用于 **MDC（Mapped Diagnostic Context）上下文增强** 的自定义注解，它的主要作用是在方法执行时自动向日志上下文中添加业务标识信息，让日志追踪更加精准。

## 14.1. 一、核心作用（通俗解释）

生活类比：快递追踪系统

想象一下你寄快递：

```
普通日志（没有 @MdcDot）：
2024-05-18 12:00:00 [INFO] 用户发布了文章
2024-05-18 12:00:01 [INFO] 用户收藏了文章
2024-05-18 12:00:02 [INFO] 用户删除了文章
↑ 不知道是哪篇文章！

加 @MdcDot 后的日志：
2024-05-18 12:00:00 [INFO] [文章ID=1001] 用户发布了文章
2024-05-18 12:00:01 [INFO] [文章ID=1001] 用户收藏了文章
2024-05-18 12:00:02 [INFO] [文章ID=1001] 用户删除了文章
↑ 知道都是对文章1001的操作，可以串联起来了！
```

## 14.2. 二、注解结构分析

```

@Target({ElementType.METHOD, ElementType.TYPE})  // 可以标注在方法或类上
@Retention(RetentionPolicy.RUNTIME)              // 运行时保留，可通过反射获取
@Documented                                      // 生成JavaDoc时包含此注解
public @interface MdcDot {
    String bizCode() default "";                  // 业务标识，支持SpEL表达式
}
```

### 14.2.1. 关键属性说明

|   |   |   |
|---|---|---|
|属性|说明|示例|
|**bizCode**|业务代码标识，支持 SpEL 表达式|`"#articleId"`, `"#req.articleId"`|
|**默认值**|空字符串，表示只记录不附加业务标识|`@MdcDot`（不带参数）|

## 14.4. 四、底层实现原理

### 14.4.1. 4.1 AOP 切面处理

[MdcAspect.java](file:///D:/Workspace/Code_Projects/paicoding-main/paicoding-core/src/main/java/com/github/paicoding/forum/core/mdc/MdcAspect.java) 负责拦截带有 `@MdcDot` 注解的方法：

```
@Pointcut("@annotation(MdcDot) || @within(MdcDot)")
public void getLogAnnotation() {
    // 匹配所有带 @MdcDot 注解的方法或类
}

@Around("getLogAnnotation()")
public Object handle(ProceedingJoinPoint joinPoint) throws Throwable {
    long start = System.currentTimeMillis();
    
    // ① 执行前：提取 bizCode 并放入 MDC
    boolean hasTag = addMdcCode(joinPoint);
    
    try {
        // ② 执行目标方法
        Object ans = joinPoint.proceed();
        return ans;
    } finally {
        // ③ 执行后：记录耗时
        log.info("执行耗时: {}#{} = {}ms",
                joinPoint.getSignature().getDeclaringType().getSimpleName(),
                joinPoint.getSignature().getName(),
                System.currentTimeMillis() - start);
        
        //  清理 MDC（防止线程池复用导致数据污染）
        if (hasTag) {
            MdcUtil.reset();
        }
    }
}
```

### 14.4.2. 4.2 业务标识提取逻辑

```
private boolean addMdcCode(ProceedingJoinPoint joinPoint) {
    // ① 先尝试从方法上获取注解
    MethodSignature signature = (MethodSignature) joinPoint.getSignature();
    Method method = signature.getMethod();
    MdcDot dot = method.getAnnotation(MdcDot.class);
    
    // ② 如果方法上没有，尝试从类上获取
    if (dot == null) {
        dot = (MdcDot) joinPoint.getSignature().getDeclaringType().getAnnotation(MdcDot.class);
    }
    
    // ③ 如果找到了注解，提取 bizCode 并放入 MDC
    if (dot != null) {
        MdcUtil.add("bizCode", loadBizCode(dot.bizCode(), joinPoint));
        return true;
    }
    return false;
}
```

### 14.4.3. 4.3 SpEL 表达式解析

这是最核心的部分，解析 `"#req.articleId"` 这样的表达式：

```
private String loadBizCode(String key, ProceedingJoinPoint joinPoint) {
    if (StringUtils.isBlank(key)) {
        return "";  // 空字符串直接返回
    }
    
    // 创建 SpEL 解析上下文
    StandardEvaluationContext context = new StandardEvaluationContext();
    context.setBeanResolver(new BeanFactoryResolver(applicationContext));
    
    // 获取方法参数名和参数值
    String[] params = parameterNameDiscoverer.getParameterNames(
        ((MethodSignature) joinPoint.getSignature()).getMethod());
    Object[] args = joinPoint.getArgs();
    
    // 将参数放入上下文
    for (int i = 0; i < args.length; i++) {
        context.setVariable(params[i], args[i]);
    }
    
    // 解析表达式，例如 "#req.articleId"
    return parser.parseExpression(key).getValue(context, String.class);
}
```

## 14.5. 五、SpEL 表达式详解

### 14.5.1. 5.1 基本语法

SpEL（Spring Expression Language）是 Spring 提供的表达式语言。

```
@MdcDot(bizCode = "#articleId")          // 提取名为 articleId 的参数
@MdcDot(bizCode = "#req.articleId")      // 提取 req 参数的 articleId 属性
@MdcDot(bizCode = "#user.name")          // 提取 user 参数的 name 属性
```

### 14.5.2. 5.2 实际案例解析

```
// 案例1：简单参数
@GetMapping("article/{articleId}")
@MdcDot(bizCode = "#articleId")  // ← 直接提取路径参数
public ResVo<ArticleDTO> getArticle(@PathVariable Long articleId) {
    // ...
}

// 案例2：对象属性
@PostMapping("post")
@MdcDot(bizCode = "#req.articleId")  // ← 提取请求体的 articleId
public ResVo<Map<String, Object>> post(@RequestBody ArticlePostReq req) {
    // req = {articleId: 1001, title: "文章标题", content: "..."}
    // bizCode = "1001"
}
```

### 14.5.3. 5.3 表达式求值过程

```
表达式："#req.articleId"

步骤1：从方法参数中找到名为 "req" 的参数
  → req = ArticlePostReq(articleId=1001, title="文章标题")

步骤2：访问 req 的 articleId 属性
  → req.articleId = 1001

步骤3：将结果转为字符串
  → bizCode = "1001"

步骤4：放入 MDC
  → MDC.put("bizCode", "1001")
```

## 14.10. 十、注意事项

### 14.10.1. 1. 参数名问题

SpEL 表达式需要知道参数名，如果编译时没有保留参数名信息，会导致解析失败：

```
// 错误：参数名丢失
public ResVo post(ArticlePostReq req) {
    // SpEL 无法识别 "req" 这个参数名
}

// 正确：使用 @RequestParam 或编译时保留参数名
public ResVo post(@RequestBody ArticlePostReq req) {
    // SpEL 可以识别 "req"
}
```

**解决方案：**

- Maven 配置：`<compilerArgs>-parameters</compilerArgs>`
- 或使用注解明确参数名

### 14.10.2. 2. 线程池复用问题

```
finally {
    if (hasTag) {
        MdcUtil.reset();  // ← 必须清理，否则下一个请求会继承上一个的 bizCode
    }
}
```

### 14.10.3. 3. 性能影响

- SpEL 表达式解析有一定性能开销
- 建议只在关键业务方法上使用
- 不要滥用，避免影响性能
