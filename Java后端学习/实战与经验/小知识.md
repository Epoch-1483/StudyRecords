# 1. IDEA 中的快捷键

## 1.1. 代码，整体向左（前移）或向右（后移）缩进
- **向右缩进（增加缩进）**：  
    `Tab` 键（选中多行后按 Tab，整块代码右移一个缩进单位）
- **向左缩进（减少缩进）**：  
    `Shift + Tab`  
    （整块代码左移一个缩进单位）

💡 缩进单位通常是 2 或 4 个空格，或一个制表符（Tab），可在设置中配置。

---

## 1.2. 选中一行代码上下移动

**Ctrl + shift + (上下键)**

# 2. 约定俗成的标记（conventional comment tags）

在代码注释中，除了 `TODO` 之外，开发者社区广泛使用一些**约定俗成的标记（conventional comment tags）**来表达不同类型的提醒或状态。这些标记通常会被 IDE、静态分析工具或代码审查系统识别并高亮显示，便于团队协作和任务追踪。

以下是**常见的类似标记**及其典型用途：

---

### 1. `TODO`

- **含义**：待办事项 —— 功能尚未完成，需要后续补充。
- **示例**：

```
// TODO: 实现用户权限校验逻辑
```

---

### 2. `FIXME`

- **含义**：已知有问题，必须修复（比 `TODO` 更紧急）。
- **适用场景**：临时绕过 bug、逻辑错误、异常处理不完整等。
- **示例**：

```
// FIXME: 这里可能引发空指针异常，需增加 null 检查
```

---

### 3. `HACK`

- **含义**：使用了不优雅、脆弱或取巧的实现方式，未来应重构。
- **提示**：虽然能工作，但“不好看”或“有技术债”。
- **示例**：

```
# HACK: 为兼容旧接口，临时强制转换类型
user_id = str(user_id) if isinstance(user_id, int) else user_id
```

---

### 4. `XXX`

- **含义**：警告！此处存在潜在风险、设计缺陷或需要注意的陷阱。
- **语气**：比 `HACK` 更强烈，常表示“这里很危险”。
- **示例**：

```
// XXX: 此处未加锁，多线程下可能数据竞争！
```

---

### 5. `NOTE`

- **含义**：提供额外说明或解释，帮助理解复杂逻辑。
- **用途**：不是问题，而是重要背景信息。
- **示例**：

```
// NOTE: 根据 RFC 7231，状态码 307 表示临时重定向且不改变请求方法
```

---

### 6. `WARNING` / `WARN`

- **含义**：强调某个操作有副作用或需谨慎使用。
- **示例**：

```
# WARNING: 修改此配置会影响所有微服务，请先联系运维团队
```

---

### 7. `OPTIMIZE`（较少见，但部分团队使用）

- **含义**：性能可优化点。
- **示例**：

```
// OPTIMIZE: 当前为 O(n²) 算法，数据量大时会成为瓶颈
```

---

### ✅ 最佳实践建议：

- **统一团队规范**：约定使用哪些标记、格式（如是否加冒号、是否写负责人）。
- **避免滥用**：不要用 `TODO` 掩盖长期不维护的技术债。
- **定期清理**：在版本发布前检查并处理关键 `FIXME`/`XXX`。
- **IDE 支持**：大多数编辑器（VS Code、IntelliJ、Vim 插件等）可自定义高亮或生成报告。

# 3. **程序退出时返回的退出码（exit code）**

当然有！在操作系统（尤其是 Unix/Linux/macOS）中，**程序退出时返回的退出码（exit code）** 是一个 8 位无符号整数（0–255），用于表示程序的终止状态。不同数字代表不同含义。

下面为你系统梳理一下常见的退出码及其含义：

---

## 3.1. ✅ **标准约定**

|   |   |
|---|---|
|退出码|含义|
|**0**|**成功正常退出**（这是唯一表示“成功”的值）|
|**1–125**|**用户自定义错误**：程序自己设定的错误码，含义由开发者决定（比如 Java 中 `System.exit(1)`<br><br>）|
|**126**|命令找到了，但**无法执行**（例如权限不足）|
|**127**|**命令未找到**（比如拼错命令名）|
|**128 + N**|程序被 **信号 N** 终止（这是关键！）|

---

## 3.2. ⚠️ **由信号导致的退出码（128 + 信号编号）**

当进程被操作系统信号（signal）强制终止时，shell 会将其退出码设为 `128 + 信号编号`。

常见信号及对应退出码：

|   |   |   |   |   |
|---|---|---|---|---|
|信号名|信号编号 (N)|默认行为|退出码 = 128+N|触发方式|
|`SIGINT`|2|中断（Interrupt）|**130**|用户按 **Ctrl+C**|
|`SIGQUIT`|3|退出并生成 core dump|131|*_Ctrl+*_|
|`SIGKILL`|9|强制杀死（不可捕获）|**137**|`kill -9 <pid>`|
|`SIGTERM`|15|终止请求（可被捕获）|**143**|`kill <pid>`<br><br>（默认信号）|
|`SIGHUP`|1|挂起（常用于终端断开）|**129**|关闭终端、SSH 断开等|

💡 注意：`SIGKILL`（9）和 `SIGSTOP`（19）**不能被程序捕获或忽略**，所以无法做清理。

---

## 3.3. 🧪 Java 中常见的退出码示例

```
System.exit(0);   // 正常退出 → exit code 0
System.exit(1);   // 一般错误 → exit code 1
System.exit(-1);  // 实际变成 255（因为只取低8位，-1 mod 256 = 255）
```

如果你在 Java 程序里写 `System.exit(130)`，那**不是被 Ctrl+C 中断**，而是你主动退出并返回 130 —— 虽然数值相同，但**原因完全不同**！

🔍 区分方法：

- 如果是你按了 Ctrl+C 或点了 IDE 的停止按钮 → 是 **被信号中断**（非主动调用 exit）。
- 如果代码里写了 `System.exit(130)` → 是**主动退出**，只是碰巧用了 130 这个数字。

---

## 3.4. 🛠 如何查看退出码？（Linux/macOS/WSL）

在终端运行程序后，立即输入：

```
echo $?
```

就能看到上一个命令的退出码。

例如：

```
java MyProgram
# 按 Ctrl+C 中断
echo $?   # 输出：130
```

---

## 3.5. 📌 小结：常见退出码速查表

|   |   |
|---|---|
|退出码|可能原因|
|0|成功|
|1|通用错误（最常见自定义错误）|
|126|权限不足，无法执行|
|127|命令未找到|
|**129**|被 `SIGHUP`<br><br>终止（如 SSH 断开）|
|**130**|被 `SIGINT`<br><br>终止（**Ctrl+C**）✅|
|**131**|被 `SIGQUIT`<br><br>终止（Ctrl+\）|
|**137**|被 `SIGKILL`<br><br>强杀（`kill -9`<br><br>）|
|**143**|被 `SIGTERM`<br><br>终止（优雅终止请求）|
|255|退出码超出范围（如 `System.exit(-1)`<br><br>）|

# 4. **CentOS 7** 使用 `nmtui` 设置静态 IP

`nmtui` 是一个在终端里运行的图形化工具，操作直观，非常适合新手。

1. **启动** `**nmtui**` **工具** 在你的 CentOS 7 终端中，直接输入以下命令并回车：

```
sudo nmtui
```

2. **编辑网络连接** 在弹出的界面中，使用键盘方向键选择 **«Edit a connection»**，然后按回车键。
3. **选择并配置网卡**

- 在左侧列表中，你会看到你的网络连接，通常是 `ens33` 或类似的名称。用方向键选中它，然后按回车键进入编辑。
- 在右侧的 «IPv4 configuration» 部分，将 «Method» 从 «Automatic» 改为 **«Manual»**。
- 点击 «Show» 按钮，然后点击 «Add» 来添加 IP 地址。
- 在弹出的输入框中，填入你想固定的 IP 地址和子网掩码。例如，如果你想固定为 `192.168.179.141`，就填写：

- **Address:**`192.168.179.140/24`
- （这里的 `/24` 等同于子网掩码 `255.255.255.0`）

- 接着，在 «Gateway» 栏填写你的网关地址（通常是 `192.168.179.2` 或 `192.168.179.1`，具体取决于你的虚拟机网络设置）。
- 在 «DNS servers» 栏，可以填写公共 DNS，例如 `8.8.8.8` 或 `114.114.114.114`。

4. **保存并重启网络**

- 配置完成后，按 «OK» 保存。
- 回到主菜单，选择 «Back»，然后选择 «Quit» 退出 `nmtui`。
- 最后，在终端中重启网络服务使配置生效：

```
sudo systemctl restart NetworkManager
```

完成以上步骤后，你的虚拟机 IP 地址就会被永久固定下来，不会再变动了。

# 5. 什么是 DSL？

在后端开发中，**DSL（Domain Specific Language，领域特定语言）** 是一种专门针对某个特定业务领域或问题设计的“微型语言”。

它的核心目的不是为了像 Java、Python 那样去编写通用的系统程序，而是为了用**最简洁、最贴近自然语言**的方式，去描述和解决特定领域的复杂问题。简单来说，DSL 就是把复杂的底层实现细节隐藏起来，让开发者（甚至业务专家）只需要关注“做什么（意图）”，而不需要关心“怎么做（实现）”。

在后端领域，DSL 通常分为两大类：

## 5.1. 1. 外部 DSL (External DSL)

这类 DSL 拥有完全独立的语法、解析器和执行环境，与宿主编程语言无关。后端开发中最常见的外部 DSL 包括：

- **SQL**：专门用于数据库查询和操作。你只需要写 `SELECT * FROM Users WHERE Age > 18;`，完全不需要了解底层数据库是如何遍历和过滤数据的。
- **正则表达式 (Regex)**：专门用于字符串的模式匹配和验证。
- **OpenAPI (Swagger)**：基于 YAML 或 JSON 格式，专门用来描述和定义 HTTP 接口的规范。
- **配置文件**：如 Nginx 的配置、Dockerfile 等，都是针对特定工具的外部 DSL。

## 5.2. 2. 内部 DSL (Internal DSL)

这类 DSL 是基于现有的通用编程语言（如 Java、Groovy、Ruby）构建的。它利用了宿主语言的语法特性（如链式调用、闭包、元编程等），让代码读起来像是一种全新的、专门的语言。后端常见的内部 DSL 有：

- **REST Assured**：在 Java 中用于 API 测试的框架。它的代码读起来就像在说话：

```
given()
    .baseUri("https://api.example.com")
    .header("Content-Type", "application/json")
.when()
    .post("/login")
.then()
    .statusCode(200)
    .body("data.token", notNullValue());
```

这种“流畅接口（Fluent Interface）”的写法就是典型的内部 DSL。

- **Gradle**：基于 Groovy 或 Kotlin 的自动化构建工具，用代码来描述项目的构建逻辑和依赖关系。
- **Spring Integration / Camel**：用于定义企业级集成模式和消息流转规则的 DSL。

## 5.3. 为什么后端需要 DSL？

在后端开发中引入 DSL 主要有以下价值：

- **提升可读性与表达力**：代码即文档。DSL 能够清晰地表达业务意图，让代码读起来像在陈述业务规则。
- **屏蔽复杂细节**：将繁琐的样板代码和底层逻辑封装在 DSL 引擎内部，使用者只需关注核心业务。
- **提高开发效率**：通过高层级的抽象，开发者可以快速构建和迭代特定领域的功能（如快速生成 CRUD 页面、定义复杂的计费规则等）。
- **促进跨团队协作**：由于 DSL 贴近自然语言，业务专家或产品经理有时也能直接参与规则的编写或审核。

总结来说，后端的 DSL 就是一种**“术业有专攻”**的编程工具，它通过高度定制化的语法，帮助后端工程师更高效、更优雅地解决特定领域的难题。
