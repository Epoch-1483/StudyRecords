
## 为什么有了字符流，还需要字节流？
### 1. 处理的数据类型不同

- **字节流（Byte Stream）**  
    处理的是**原始的二进制数据**（8位字节），适用于所有类型的数据，比如图片、音频、视频、压缩文件、可执行文件等。这些数据本质上不是文本，不能直接用字符编码解释。
    
- **字符流（Character Stream）**  
    专门用于处理**文本数据**，它在底层使用字节流，但会根据指定的字符编码（如 UTF-8、GBK）将字节转换为字符（通常是 16 位的 Unicode 字符）
### 2. **字符流依赖编码，字节流不依赖**

- 字符流在读写时必须指定或默认使用某种字符编码。如果编码不匹配，文本就会乱码。
- 字节流完全不关心内容含义，只是原样传输字节，因此**更底层、更通用**。

### 3. **网络和底层 I/O 本质是字节**

- 网络协议（如 TCP/IP）、文件系统、硬件设备等，传输和存储的单位都是**字节**。
- 字符流其实是对字节流的**封装和抽象**，只适用于文本场景。
- 在开发底层库、协议解析、加密、序列化等场景中，必须直接操作字节。

### 4. **性能与控制粒度**

- 字符流为了处理编码，会做额外的转换工作，有一定开销。
- 如果你不需要字符处理（比如只是复制文件），用字节流更高效、更直接。

## 字符流原理解析：

1.创建输入流对象其实就是在数据源(UTF-8) 和 内存中建立了一个通道
```
FileReader fr = new FileReader("myio\\a.txt");  
```
底层：
		关联文件，并创建缓冲区（长度为8912的字节数组）

2.读取数据：
内存中有缓冲区长度为8192的字节数组
每次都从缓冲区读取数据这样做能提高效率
缓冲区没有就从文件中读取数据，尽可能装满缓冲区

## flush 和 close方法：
![[Pasted image 20251028133854.png]]

close : 断开通道，无法再往文件中写出数据

```
FileWriter fw = new FileWriter("myio\\a.txt");  
for (int i = 0; i < 8192; i++) {  
    fw.write(97);  
}  
System.out.println("==================");
```
超过8192个字节后自动保存到文件中

```
fw.write("小小小小小");  
fw.write("说话很好听");  
fw.flush();  
// 下面的代码不会执行，但是可以继续写数据  
fw.write("你好你好你好");  
fw.write("再见再见再见");  
// 如果是close的话下面写数据会直接报错  
fw.close();
```
## 加密存储和解密
异或  ^ 两边是数字的话会进行二进制运算
解密把girl.jpg 改为ency.jpg 然后把ency.jpg改为其他就可以看到解密文件
```
// 解密把girl.jpg 改为ency.jpg 然后把ency.jpg改为其他就可以看到解密文件
FileInputStream fis = new FileInputStream("myio\\girl.jpg");  
FileOutputStream fos = new FileOutputStream("myio\\ency.jpg");  
  
int b;  
while ((b = fis.read()) != -1){  
    fos.write(b ^ 2);  
}  
fos.close();  
fis.close();
```
